## Слайд 3: Практика: Liquibase Cloud-to-Local
**Текст:**
1.  **Связь:** Используем JDBC строку из настроек Supabase.
2.  **Snapshot:** 
    ```bash
    liquibase --url=jdbc:postgresql://db.[REF].supabase.co:5432/postgres \
              --username=postgres --password=[PASS] \
              generate-changelog --changelog-file=db/initial.xml
    ```
3.  **Sync:** Накат на локальный Supabase:
    ```bash
    liquibase --url=jdbc:postgresql://localhost:54322/postgres update
    ```
**Визуал:** Скриншот XML-файла чейнджлога и лога выполнения команды `update`.

> **Заметки преподавателя:** Смотрите, Liquibase сам проанализировал нашу облачную базу и создал XML-описание всех таблиц и связей. Теперь, если мы захотим развернуть проект локально или на новом сервере, нам не нужно ничего "кликать" в UI — Liquibase сделает всё за нас. Это и есть Database-as-Code.

## Техническое приложение (Для преподавателя)

### 1. Настройка Liquibase для Supabase
Для работы потребуется файл `liquibase.properties`:
```properties
changeLogFile: db/changelog.xml
url: jdbc:postgresql://db.[REF].supabase.co:6543/postgres?ssl=true&sslmode=require
username: postgres
password: [YOUR_PASSWORD]
driver: org.postgresql.Driver
```
*Примечание: Используйте порт 6543 (Transaction Pooler) для стабильности в CI.*

### 2. Реализация Log Reader (Option A)
Скрипт-заглушка для агента (может быть оформлен как MCP):
```python
# get_logs.py
import subprocess

def get_supabase_logs(project_id, service="db"):
    # Команда CLI для получения логов
    cmd = f"supabase logs read --project-ref {project_id} --service {service} -n 20"
    result = subprocess.run(cmd, shell=True, capture_output=True, text=True)
    return result.stdout

# Агент вызывает это через run_shell_command или MCP
```

3. Просим агента найти ошибку, используя доступ к логам.
